# 数创智伴 · 问答/操作模式拆分设计

**日期**：2026-07-20
**作者**：Codex + 用户共创
**状态**：设计草案 v1（待用户审阅）
**范围**：前端 `PageAgentPanel` UI 改造 + 数据流微调；不改后端、不改 page-agent 库、不改 FAB

---

## 1. 问题陈述

现状 `PageAgentPanel` 提供两个并列提交按钮 —— **"问当前页"** 与 **"执行操作"** —— 但共用同一个输入框与同一条对话历史。用户需要根据自己想做的事**主动判断**该按哪个按钮，导致：

1. **意图识别负担前置给用户**：输入"这篇文章在讲什么？"按"执行操作"会被 page-agent 当成操作任务跑 DOM，反而给出误导结果。
2. **上下文污染**：问答内容与 page-agent 的"执行步骤回执"挤在同一条 history，刷新页面、跨路由切换后旧消息互相干扰。
3. **意图歧义**：用户描述模糊时（如"这个报告"），两个按钮触发完全不同的链路，结果不可预期。

目标：**把"选模式"这件事从用户脑子里转移到 UI 上**，所见即所选，无需前置意图识别。

---

## 2. 设计决策

### 2.1 模式入口：顶部 Tab

在面板 `contextBar`（页面上下文条）下方、`body`（聊天区）上方新增一段 **模式切换 Tab**，包含两个 Tab：

| Tab | 文案 | 图标 | 默认色 |
|---|---|---|---|
| 读懂本页 | `BookOpen` | 蓝（#315fae） | 否（默认未激活） |
| 协助操作 | `MousePointerClick` | 金（#b9963f） | 否（默认未激活） |

**激活态样式**：深色实心背景（`#18233b`）+ 白字；未激活态透明 + 灰字。

**默认 Tab**：`读懂本页`（决策 1.A）。理由：绝大多数新用户首次进入是"想了解当前页内容"，操作模式是进阶用法。

**Tab 切换时**：保留两边的 history 不清空（决策 2.A）；只切换显示过滤。理由：避免来回切模式丢失上下文。

### 2.2 历史分轨：两条独立 history

```ts
type UiMessage = {
  id: number
  role: 'user' | 'assistant'
  content: string
  mode: 'ask' | 'operate'  // 新增
  routeKey?: string        // 现有，按路由分桶
}
```

- 切换 Tab 仅切换 **显示**，不修改存储
- `sessionStorage` key 改为 `hbsc.page-agent.chat.history:<routeKey>:<mode>`，与 mode 绑定
- **清空按钮**：从单按钮变为两按钮 — 「清空问答」「清空操作」；或保留单按钮但弹层里二选一
  - **决策**：单按钮 + 弹层二选一（避免 UI 拥挤）

### 2.3 输入框与提交按钮

- **单一提交按钮**（合并现有"问当前页"/"执行操作"两按钮），按钮文案 + 图标跟随当前 Tab 变化：
  - 读懂本页 Tab 激活时：按钮 = `📖 提问`
  - 协助操作 Tab 激活时：按钮 = `🖱 执行`
- **placeholder 跟随模式变化**：
  - 读懂本页：`问一个关于本页的问题…`
  - 协助操作：`描述想让我做的事，如「跳到搜索页」…`
- **快捷键**（决策 3.A）：Enter 发送；Shift+Enter 插入换行

### 2.4 气泡视觉差异化

为了在历史混看时清晰区分两种模式，AI 气泡顶部追加一个小标签：

| 模式 | 标签色块 |
|---|---|
| 读懂本页 | 浅蓝 `#edf3ff` 文字 `#315da9` |
| 协助操作 | 浅金 `#fff5e0` 文字 `#8a6618` + 金色边框 |

用户气泡保持原样不变。

### 2.5 范围之外（明确不做）

- **不改 FAB**（`PageAgentFab`）— 保持单浮窗入口
- **不引入意图识别**（不调 LLM 判断模式）— 纯 Tab 显式切换
- **不改后端接口** — `/api/public/agent/execute` 与 `/api/public/agent/llm` 调用契约不变
- **不改 page-agent 库** — `customFetch` / `getPageHint` / `maskSecrets` 不动
- **不做快捷键强制切 Tab**（如 Cmd+1/2）— 留作未来增强

## 3. 数据流

### 3.1 状态变化（`PageAgentPanel.tsx`）

**新增 state**：
```ts
const [mode, setMode] = useState<'ask' | 'operate'>('ask')
```

**修改 state**：
```ts
// UiMessage 新增 mode 字段
type UiMessage = { id: number; role: ...; content: string; mode: 'ask' | 'operate'; routeKey?: string }
```

### 3.2 存储 key 变更

```
旧: hbsc.page-agent.chat.history:<routeKey>
新: hbsc.page-agent.chat.history:<routeKey>:ask
    hbsc.page-agent.chat.history:<routeKey>:operate
```

**迁移策略**：第一次加载时若发现旧 key 存在：
- 内容里 mode 字段缺失的旧消息 → 默认归为 `ask`（保守起见，绝大多数旧用法都是问答）
- 迁移完成后删除旧 key
- **已知局限**：旧历史里如果混有"执行操作"内容，会被归入 ask 桶显示。这是可接受的——用户可在清空弹层手动归档（v1.1 可加"重新归类"动作，本次不做）

### 3.3 调用分发

```ts
async function send() {
  // ... 校验
  if (mode === 'ask') return sendAsk()
  return sendOperate()
}
```

`sendAsk` / `sendOperate` 内部逻辑保留；只是把"按哪个按钮"换成"由 mode state 决定"。

### 3.4 清空弹层

保留单按钮 → 弹层改成两段：
```
┌───────────────────────────────┐
│ 清空本次对话？                 │
│ 选择要清空的范围：              │
│  [ ] 清空问答 (3 条)          │
│  [ ] 清空操作 (1 条)          │
│       [取消]  [确认清空]      │
└───────────────────────────────┘
```

未勾选任何项时确认按钮 disabled。

## 5. 验收标准

1. **视觉**：进入面板默认 Tab = 读懂本页；切换 Tab 输入框 placeholder / 提交按钮文案 / 颜色同步变化
2. **数据**：在读懂本页 Tab 下输入三条问答，切到协助操作 Tab 再输入两条操作；切回读懂本页，三条问答仍完整存在；切到协助操作，两条操作仍完整存在
3. **持久化**：刷新页面后，两条 history 各自按 (routeKey, mode) 恢复
4. **清空**：清空弹层支持独立勾选问答/操作；勾选一个 + 确认 → 只清空对应桶，另一桶历史不受影响
5. **回归**：现有后端调用契约不变（`sendAsk` 仍调 `/execute`，`sendOperate` 仍走 `page-agent.execute`）
6. **兼容性**：旧 sessionStorage 数据第一次加载自动迁移；无旧数据的新用户零额外迁移
7. **预览页清理**：方案落地后删除 `/labs/agent-preview` 预览路由与 `AgentPreviewLab.tsx` / `agent-preview.css`（决策点：在写实施计划时由用户确认是否一次性删除）

---

## 6. 风险与缓解

| 风险 | 缓解 |
|---|---|
| 用户没注意顶部 Tab，导致旧习惯（找按钮）失效 | 输入框 placeholder 和提交按钮文案明确区分；空态提示文案强调"切换 Tab" |
| 迁移丢失用户历史 | 迁移前在 console 留 log；保守默认归为 ask |
| Tab 切换动画造成性能问题 | 仅做 150ms 颜色过渡；不重新挂载子组件 |

---

## 7. 验收后动作

- 删除 `/labs/agent-preview` 预览页（确认方案后已无价值）
- 如需保留做回归对照，则将 `AgentPreviewLab` 组件改为只展示最终方案

## 附录 A：与方案 B/C 的对比回顾

| 维度 | A 模式先行 | B 双入口 | C 智能默认 |
|---|---|---|---|
| 模式判断 | 用户显式 | 用户显式 | 启发式（前端正则） |
| UI 复杂度 | 中（多一层 Tab） | 低（FAB 变宽，面板简化） | 低（单提交） |
| 会话隔离 | 干净（分 key） | 干净（入口隔离） | 弱（共享 history） |
| 误判风险 | 0 | 0 | 有 |
| 视觉一致性 | 高（FAB 不动） | 中（FAB 形态变化） | 高 |

**选 A 的核心理由**：把"决策"留给用户最自然的位置（看到选项），同时保留视觉一致性。
